SOLID & Design Patterns

SOLID is an acronym for 5 principles to guide engineers in writing good code. The name SOLID was coined by Michael Feathers and promoted by Robert Martin (aka "Uncle Bob"), two very prominent figures in the world of computing. These principles establish practices that help develop applications that are easier to maintain over time and as the code base grows.

5 Principles of SOLID

     S - Single-responsibiltiy
     O - Open/Closed
     L - Liskov Substitution
     I - Interface Segragation
     D - Dependency Inversion

Applying SOLID

Single-Responsibility Principle states that a class should only have one job. This makes code readability and maintenance easy by logically organizing the code by its function. We can see this in our operations classes, where each of the arithmetic operations are organized into its own class.

Open-Closed Principle is basically stating that our code should exercise encapsulation, which we discussed in our OOP page. The idea is that our code should be open to use but not so accessible to edits. By coding in this way, our application is less prone to break when a feature needs to be updated or changed. To review these concpets please navigate to the OOP page or click on the blue link.

Liskov Substitution Principle communicates that every subclass should be able to substitute for their parent class. By employing inheritance, this principle is automatically employed, where the methods and attributes defined in the Calculations class are accessible when one of its subclass, like Division is instantiated. To see the screenshots, please click on the blue link above and go to the Inheritance section.

Interface Segregation Principle informs us that a client should never be forced to implement a part of the program it does not use. In another words, if we built a calculator application that ony adds and subtracts, then it should not bundle in the multiplication and division attributes. By creating a separate class for each of these operations, we can avoid this situation, whereby our limited calculator will never have to instantiate the classes it does not need.

Dependency Inversion Principle encourages us to rely on abstraction so that there are no hard dependencies on a piece of code. This ensures that our code is stable when functionality is edited, but flexible in that our code allows for such changes without difficulty. We have employed this in our code by having the divide method abstracted away from the Division class. This ensures that changes to the Division object will not require changes to the way we divide numbers in our code.

Design Patterns

A design pattern is a general solution for common application design problems and is meant to make development faster, while making your code agile and maintainable. There are 3 main types of design patterns:
Creational patterns are used for object creation and helps develop code that employs abstraction and helps object instantiation be efficient.
Structural patterns help in organizing classes to form larger structures and provide new functionality.
Behavioral patterns provide a template for creating applications that rely on communication between objects. This pattern is more for managing algorithms and relationships between objects
In our Calculator program, a creational pattern best suits our needs because, every time a user inputs an arithmetic operation, we need an efficient way to manage those operations. The creational pattern informs us to build a class for each of our operations and to use abstraction, encapsulation and polymorphism to efficiently manage our operation instantiations.

Of the creational patterns, the Factory Method suits our needs best and is what I just described above. In Factory pattern, objects are instantiated without revealing the creation logic, while a client uses one interface to create new objects. In our calculator program, a client uses the Calculator object to manage all Calculations and Operations objects.
