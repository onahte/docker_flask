OOP

Object oriented programming or OOP is a programming paradigm, where programs are organized as objects. An object's behaviors and characteristics are defined by a class and its methods. Below is a glossary of terms and an explanation of how OOP principles were applied in our basic calculator app.

constructor
a required method in a class that defines the characteristics of an object

instance method
a class method that defines an attribute or action of an object

instance property
an object property that's defined by the class constructor

static method
method that is bound to the class and not an instance

static property
use of keyword static to denote a class property

namespace
a rule where a variable within a domain cannot have the same name else a name collision will occur

fixture
code testing feature where a fixed environment is created so that test results are repeatable

type hint
a way for the programmer to hint at the type of object being used

type cast
to change a variable from one data type to another

Glossary

object
basic unit of OOP where both the data and function that operate on data are bundled as a unit

class
template that describe a set of properties that belong to an object

instantiation
process of creating an object or an instance of a class

static
an element of the program that cannot be instantiated

encapsulation
practice of binding object features into a single unit and restricting access to certain properties

inheritance
practice of creating a new child class from an existing class, thereby preventing code duplication but custom application of an existing class

polymorphism
practice of structuring the program so that calls to a method will return the right method call despite other classes of the same family using the same method name

abstraction
practice of allowing only essential information to be accessible and hiding the background details

Encapsulation & Abstraction

Our calculator program begins by defining the calculator operations: addition, subtraction, multiplication and division. For conciseness, I have shown just the Division class on the right but note that the following discussion applies for all classes. Division's static method divide returns the value after dividing value_1 by value_2 . This is a static method because no object needs to be instantiated to run divide . To divide two values, our Calculator class calls Division class's divide method, which also helps abstract away the code used to divide. You can see a Calculator object instantiated in our test_calculator_divide_method method.

Inheritance & Polymorphism

Using our Division class again as an example, we can see inheritance in action. In another words, the Division class inherits all the methods and attributes of the Calculation class, which allows us to minimize code repetition. That's why we are able to use the create method of the Calculations class without having to write it in the Division class code.

Our Division class also employs polymorphism; by defining the get_result method in all our classes, the get_result call will "smartly" retrieve the right get_result.
