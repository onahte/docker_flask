Heroku

Heroku is a PaaS that's a part of our automation pipeline. The Heroku platform is built on top of AWS but the cloud-level configurations have been abstracted away and done for us by Heroku. Furthermore, Heroku allows us to test our deployment by defaulting the deploy as https://<APP_NAME>.herokuapp.com. There is of course, the option to deploy the service with a personal domain.

Automation & YAML

As mentioned in the Git section of this project site, this website is automatically instantiated by Github via the prod.yml and dev.yml Workflow files. These YAML files provide the metadata for Github to know what and how to execute the automation (so using Github's action keywords is very important). Github will execute the Workflow file in the order that it was composed. Simple boolean logic is built-in but please note that YAML is a mark-up language and not a programming language, so it is not capable of complex logic (I learned this the hard way).

The prod.yml takes advantage of PyTest to test our page renders as well as the akhileshns/heroku-deploy repo to provide the code to launch our production Heroku app. There is no testing executed by dev.yml but like prod.yml, we take advantage of the akhileshns/ heroku-deploy repo to deploy the development Heroku app. The Docker image of this project is deployed from the Heroku app, which is a container of the Python Blueprint and the Flask framework and they in turn, render this website. All this is triggered by a git push. It's really quite remarkable.
