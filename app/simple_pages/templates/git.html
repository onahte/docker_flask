Git

Git is a versioning tool that allows multiple collaborators to work together on a project while minimizing code conflicts. Remote versions of the main code can be created and worked on independently. Then, each collaborator's version (or branch) can be uploaded to the code base and merged after review. Github is a website that gives the public access to the Git technology and can house your code base as repositories (aka repo). Github also offers many other tools that make code development and production easier and more secure.

Repos & Branches

Github allows you to work directly in your repo or clone a repo to allow you to work remotely on your personal workspace. It's always a good idea though, to branch off of the main branch and create a child branch before you begin. This is so that potentially problematic code can be isolated to your child branch and your main code base can be preserved. As you are working, updates to the repository can be made via the add, commit and push commands. git add <FILE> stages the revision you would like to upload to your repo. git commit snapshots the staged revisions and git push -u <DESTINATION REPO> <BRANCH> uploads those revisions to your repo. Be sure to pass the -m flag with your commits with a meaningful description of the revision being made. This is because, Github keeps a history of commits, which you and your team can reference to troubleshoot, make changes or roll back revisions. An informative description helps everyone. To see the revision history, type in the command git log.

Tagging & Merging

Git's tag feature allows you to mark a certain point in your branch history. You may want to tag a commit as a progress marker or to note a version release of your code. A tag is meant to be permanent and can be performed with the git tag -l "<TAG>" command. To see information of a tagged commit, use the git show <TAG> command.

Merging, as mentioned before can be implemented when the code of any two branches are ready to be combined. To do so, run git merge <BRANCH_NAME> from the branch you want to merge into and passing into the command the branch you want to merge with. The branches will merge with the latest commit in the branch's respective revision history.

Automation: Secret & Actions

Github offers a feature called Actions (also accessible from within the repo) that helps you automate your CI/CD pipeline. By navigating to the Actions page, you can see the actions history and their pass/success status. By clicking on an action, you can find more detailed information in case you need to troubleshoot. These actions are triggered by implementing Workflow files, which needs to be organized into .github/workflows directory structure in the repo. For this project, the git push to the main branch triggers the prod.yml file and similarly, the git pull to the main branch triggers the dev.yml file. Each of these files contain the task list to be automated. I dive deeper into these Workflow files in the CI/CD section of this website. Please feel free to skip ahead by clicking on the blue link.

By navigating to the Settings page of the repo, the Secret feature allows you to enter in the login credentials of the accounts used as part of the CI/CD pipeline. This feature allows the developer to safely store the credentials without exposing them in plain text in the Workflow files. As you can see for this project, I have my Dockerhub and Heroku accounts set up as part of my automated CI/CD pipeline.
